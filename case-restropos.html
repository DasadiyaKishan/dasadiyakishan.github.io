// Restropos POS interface · 16:9

Problem

Restaurants needed a POS that the floor staff could learn in an afternoon, and that owners could read without calling a developer. The previous system mixed three roles into one screen — and locked all of them when one was busy.

What I built

API for the tablet client

REST endpoints for orders, items, modifiers, splits, kitchen events, and settlements. Idempotent writes, optimistic UI on the client side.

Owner admin

A Bootstrap dashboard for menu, taxes, staff, and the day's report. Boring forms, fast pages, no surprises.

Reports that match the till

Daily and shift-level reports calculated from the same source the receipt prints from. Owners stop arguing with the system on day one.

Outcome

Rolled out across the founder's restaurants first, then sold to other owners. Kitchen never went down during peak hours.
